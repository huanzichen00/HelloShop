# Interview QA

## 1. 这个项目是做什么的？

这是一个基于 Spring Boot、MyBatis、MySQL 和 Redis 的后端练习项目，我主要用它系统训练 Java 后端开发中的常见能力。项目目前包含 `users` 和 `tasks` 两个业务模块，已经实现了基础 CRUD、参数校验、统一返回、全局异常处理、分页、排序、多条件动态查询和事务实验，这两个模块之间也已经做了基础关联查询。另外，我还补了注册、用户名密码登录、JWT 鉴权、Redis 黑名单 logout、当前用户资料维护、任务模块的数据权限控制、任务状态流转和完成时间自动维护、任务状态统计接口，以及任务状态变更记录和日志查询。

我做这个项目的目标不是只把接口写出来，而是把后端开发里比较常见的一整套能力串起来，形成一个能讲、能继续扩展的小型后端项目。

## 2. 你在项目里主要负责了什么？

这个项目目前是我自己主导完成的。我主要做了这些事情：

- 设计了 `users` 和 `tasks` 两个模块的表结构和接口范围
- 用 Spring Boot 搭建项目结构

- 用 MyBatis 注解版和 XML 版实现数据库访问

- 做了分页、排序、多条件动态查询
- 做了用户和任务两个模块之间的基础关联查询
- 做了注册、用户名密码登录、JWT 鉴权、Redis 黑名单 logout、当前用户资料维护和任务模块的数据权限控制

## 3. 为什么要做统一返回结构？

如果接口有的直接返回对象，有的返回字符串，有的返回错误页面，那前端处理起来会很混乱。  
所以我在项目里统一用了 `Result<T>` 来包装普通接口返回，用 `PageResult<T>` 来包装分页接口返回。

这样做的好处有三个：

- 前后端约定更稳定
- 成功和失败结果结构统一
- 后续做异常处理、接口文档和项目讲解都更清晰

## 4. 为什么要做全局异常处理？

如果没有全局异常处理，项目里一旦抛异常，前端通常只能收到 Spring 默认的错误响应，结构不统一，也不利于前端识别具体错误原因。

所以我做了：

- `BusinessException`
- `ErrorCode`
- `GlobalExceptionHandler`

这样业务异常和参数校验异常都可以统一收口成自己的 JSON 返回，比如：

```json
{
  "code": 404,
  "message": "task not found",
  "data": null
}
```

这样比默认错误页面更适合实际开发。

## 5. 为什么 MyBatis 有的地方用注解版，有的地方用 XML？

我没有一开始就全部用 XML，而是按场景来分。

简单固定 SQL，比如基础 CRUD，我用注解版，因为写起来直接、清晰。

复杂动态 SQL，比如：

- 多条件查询
- 分页
- 排序
- 动态 `where`

我切到了 XML，因为 XML 更适合表达 `<if>`、`<where>`、`<choose>` 这种动态逻辑。

所以我对两种写法的理解是：

- 注解版适合简单固定 SQL
- XML 更适合复杂动态 SQL

## 6. 你项目里的分页是怎么做的？

我做分页时，不是只查当前页数据，而是分成两步：

1. 查当前页的列表数据
2. 查满足条件的总条数

然后把结果组装成 `PageResult<T>`，里面包括：

- `list`
- `total`
- `page`
- `size`
- `totalPages`

其中 `offset` 的计算方式是：

```text
offset = (page - 1) * size
```

总页数是通过向上取整公式算出来的。

## 7. 你对事务做了哪些实验？结论是什么？

我在项目里专门做了几组事务实验，重点验证了三个问题。

第一，`RuntimeException` 默认会回滚。  
第二，checked exception 默认不会回滚，除非显式写 `rollbackFor = Exception.class`。  
第三，同一个类内部直接调用带 `@Transactional` 的方法时，事务可能失效，因为这类调用绕过了 Spring 代理。

我不仅看了接口结果，还看了事务管理器日志里的 `commit` 和 `rollback`，这样能更准确判断事务最终行为。
