# Spring Boot 学习笔记 02

## 1. 本节目标

这一节不再只关注“接口能不能跑”，而是开始建立最基础的分层意识：

1. `Controller` 负责什么。
2. `Service` 负责什么。
3. 为什么不要把所有逻辑都堆在 Controller 里。

## 2. 当前项目结构

当前项目已经有了最小的分层雏形：

- `controller`
- `service`

当前理解先记成这样：

- `controller`：接收请求、返回响应
- `service`：处理业务逻辑

## 3. 为什么要分层

如果把所有代码都写在 `Controller` 里，短期看起来快，长期会很乱。

问题主要有三个：

1. 请求接收和业务逻辑混在一起，不容易维护。
2. 后面逻辑一多，Controller 会迅速膨胀。
3. 不利于测试，也不利于以后接数据库。

所以从现在开始要形成一个最基础的习惯：

- Controller 不直接承担全部业务逻辑。
- 业务逻辑先往 Service 里放。

## 4. 当前代码里的分工

### `HelloController`

现在负责：

1. 定义接口路径
2. 接收参数
3. 调用 `UserService`
4. 把结果返回给前端

### `UserService`

现在负责：

1. 处理“创建用户”
2. 处理“查询用户列表”

注意：

当前 `UserService` 里还是内存中的 `List`，并没有接数据库。

这是故意的。

原因是当前学习重点是：

1. 先理解分层
2. 先理解请求和响应
3. 暂时不引入 MySQL 和 MyBatis 的复杂度

## 5. 这一节你应该真正掌握什么

1. `Controller` 不是“所有逻辑都往里写”的地方。
2. `Service` 是业务逻辑的承接层。
3. 分层不是为了形式，而是为了让项目能继续长大。
4. 即使还没接数据库，也可以先把结构搭对。

## 6. 当前的直觉模型

你现在可以把一次请求先想成这条链路：

`浏览器 / Postman -> Controller -> Service -> 返回结果`

后面接数据库后，这条链路会继续变成：

`浏览器 / Postman -> Controller -> Service -> Mapper / Repository -> 数据库`
