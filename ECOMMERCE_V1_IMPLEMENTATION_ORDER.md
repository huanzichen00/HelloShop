# SpringBoot Hello 电商系统 V1 开发顺序说明

## 1. 目标

这份文档用于明确电商系统第一版真正开始实现时，应该按什么顺序推进，避免一开始同时铺太多模块，导致项目失控。

当前核心原则仍然是：

```text
先身份
再内容
再过程
最后结果
```

## 2. 第一阶段：先整理现有学习项目底座

在进入电商业务开发之前，先做一次低风险整理。

### 先做什么

1. 收口旧的 `users/tasks` 学习代码和演示接口边界
2. 修复明显的测试和代码问题
3. 把数据库配置改成更安全的本地覆盖方式
4. 把电商第一版文档定下来

### 这一阶段的完成标准

- 项目还能正常启动
- 现有测试状态清楚
- 电商改造范围清楚

### 这阶段先不要做什么

- 不要一开始就建所有电商表
- 不要先引入 Spring Cloud
- 不要先加支付、地址、物流

## 3. 第二阶段：项目骨架

### 先做什么

1. 明确新包结构
2. 准备电商相关 DTO 目录
3. 准备电商 model / mapper / service / controller 占位
4. 如果需要，补充新的分页查询 DTO

### 完成标准

- 目录结构已经为电商主线准备好
- 新模块命名已经固定

## 4. 第三阶段：认证主线

### 先建哪张表

先立住：

```text
users
```

### 先做哪些接口

1. `POST /auth/register`
2. `POST /auth/login`
3. `GET /auth/me`

### 这一阶段要补的能力

- 用户实体统一
- 用户注册逻辑
- 用户登录逻辑
- JWT
- 当前用户识别

### 测试重点

- 注册成功
- 登录成功
- 未登录访问受保护接口返回未登录
- 当前用户接口能正确返回

## 5. 第四阶段：商品内容主线

### 先建哪张表

推荐顺序：

1. `categories`
2. `products`

### 先做哪些接口

1. `GET /categories`
2. `GET /products`
3. `GET /products/{id}`

### 这一阶段测试重点

- 分类列表能返回
- 商品分页列表能返回
- 商品详情能正确返回
- 商品状态和库存字段能正确映射

## 6. 第五阶段：购物车主线

### 先建哪张表

```text
cart_items
```

### 先做哪些接口

1. `GET /cart/items`
2. `POST /cart/items`
3. `PUT /cart/items/{id}`
4. `DELETE /cart/items/{id}`

### 这一阶段测试重点

- 只能操作自己的购物车项
- 重复加入同一商品时是否合并数量
- 商品下架或不存在时不能加入购物车
- 数量必须合法

## 7. 第六阶段：订单主线

### 先建哪张表

推荐顺序：

1. `orders`
2. `order_items`

### 先做哪些接口

1. `POST /orders`
2. `GET /orders`
3. `GET /orders/{id}`

### 这一阶段要重点解决的问题

- 订单号生成
- 订单金额计算
- 订单项快照
- 扣库存
- 清理已下单购物车项
- 事务边界

### 这一阶段测试重点

- 购物车为空不能下单
- 库存不足不能下单
- 下单成功后生成订单和订单项
- 下单成功后库存减少
- 用户只能查看自己的订单

## 8. 第七阶段：第二批增强

第一版主线稳定后，再按优先级补：

1. `PUT /users/me`
2. `POST /orders/{id}/cancel`
3. 后台商品 CRUD
4. 后台分类 CRUD
5. 接口文档补充
6. 更完整的测试数据

## 9. 当前为什么不马上上 Spring Cloud

因为现在如果直接拆微服务，会让本来该练的业务问题被基础设施问题盖住。

当前更合理的顺序是：

1. 先做模块化单体
2. 把上下文边界保持清楚
3. 等单体闭环稳定后再拆服务

## 10. 后续微服务拆分顺序建议

如果电商 V1 单体完成，后面想上 Spring Cloud，建议顺序是：

1. 先拆 `user-service`
2. 再拆 `product-service`
3. 再拆 `order-service`
4. `cart-service` 可根据体量决定是否单独拆
5. 最后再引入 `gateway-service`

原因是：

- 用户和商品通常边界最清楚
- 订单逻辑最复杂，应该在单体里先做稳
- 购物车体量较小时可以先保留在单体或并入订单域

## 11. 当前这一轮最推荐的实际落地顺序

如果从下一步代码实现开始，最推荐的顺序是：

1. 修底座和测试
2. 建电商包结构
3. 先做 `categories + products`
4. 再做 `cart_items`
5. 最后做 `orders + order_items`

这样推进节奏最稳。
