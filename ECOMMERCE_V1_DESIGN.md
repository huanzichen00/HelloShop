# SpringBoot Hello 电商系统 V1 设计说明

## 1. 目标

这份文档用于明确 `springboot-hello` 从学习项目升级成电商系统时，第一版到底做什么、不做什么，以及为什么这样切范围。

当前目标不是直接做成“大而全”的商城，更不是一开始就强行拆成微服务，而是先做一个：

- 可运行
- 可讲清楚
- 可继续扩展
- 后面能平滑演进到 Spring Cloud 微服务

的电商后端第一版。

## 2. 当前项目为什么适合改成电商系统

你现在这个学习项目已经具备了不少可复用基础：

- 统一返回结构
- 全局异常处理
- DTO 参数校验
- MyBatis + MySQL
- JWT 鉴权
- Redis 黑名单 logout
- 当前用户识别
- 基础数据权限控制
- 分页和条件查询

这说明下一步不应该继续停留在 `users + tasks` 练习模型，而应该进入：

**围绕真实业务做建模和流程设计**

电商系统是一个很合适的方向，因为它能系统练到：

- 表关系设计
- 事务边界
- 状态流转
- 快照思维
- 数据权限
- 后续微服务拆分边界

## 3. 第一版项目定位

第一版定位为：

**单体架构下的电商后端最小闭环**

当前最核心要解决的问题是：

- 用户可以注册和登录
- 用户可以浏览商品分类和商品
- 用户可以维护购物车
- 用户可以提交订单
- 用户可以查看自己的订单

也就是先把：

```text
身份体系 + 商品浏览 + 购物车 + 下单 + 我的订单
```

这条主线做稳。

## 4. 第一版技术路线

### 后端主线

- Spring Boot
- MyBatis
- MySQL
- Redis
- JWT
- Swagger / SpringDoc
- Jakarta Validation
- 统一返回结构
- 全局异常处理

### 当前明确不做

- Spring Security
- Spring Data JPA
- 消息队列
- 分布式事务
- 注册中心
- 配置中心
- 服务网关

原因不是这些技术不重要，而是当前更重要的是：

**先把业务主线做稳，再为微服务拆分留边界。**

## 5. 第一版最小模块

第一版建议固定成 5 个核心模块。

### 1. 用户认证模块

用于解决：

- 注册
- 登录
- 当前用户识别

### 2. 分类模块

用于解决商品如何组织和展示。

### 3. 商品模块

作为内容主资源，解决用户“逛什么、看什么、买什么”。

### 4. 购物车模块

用于承载用户的待购买商品集合，是下单前的过渡区域。

### 5. 订单模块

用于承载一次购买行为的结果，包括订单主表和订单项明细。

## 6. 第一版明确暂不实现

第一版先不做：

- 支付接入
- 优惠券
- 地址簿
- 发货物流
- 评价系统
- 秒杀
- 搜索引擎
- 推荐系统
- 后台运营报表
- 图片上传
- 管理员 RBAC

这些都可以作为第二阶段或微服务化之后的扩展项。

## 7. 第一版最小业务闭环

第一版真正要立住的业务闭环是：

1. 用户注册
2. 用户登录
3. 浏览分类和商品
4. 把商品加入购物车
5. 调整购物车数量
6. 提交订单
7. 查看我的订单列表
8. 查看订单详情

如果这条闭环能稳定跑通，这个项目就已经从“后端练习项目”进入“可展示的小型业务系统”。

## 8. 核心资源判断

### 行为主体

- `user`

### 内容主资源

- `category`
- `product`

### 过程型资源

- `cart_item`

### 结果型资源

- `orders`
- `order_item`

这个划分很重要，因为它决定了后面开发顺序应该是：

```text
先身份
再内容
再过程
最后结果
```

## 9. 关键业务规则

### 1. 购物车项属于用户

用户只能查看和操作自己的购物车项。

### 2. 下单时生成订单快照

`order_item` 里不只存 `product_id`，还要保存：

- 商品名
- 下单时单价
- 下单数量

这样即使商品后面改名或改价，历史订单仍然能正确展示。

### 3. 订单金额由后端计算

不能信任前端直接传总价。

### 4. 下单时要校验库存

库存不足时不能创建订单。

### 5. 库存扣减和订单创建要放在同一个事务里

这会成为电商系统里很重要的一条事务主线。

### 6. 订单有明确状态

第一版建议先保留最小状态集：

- `CREATED`
- `CANCELLED`

如果后面要补支付，再扩展：

- `PAID`
- `SHIPPED`
- `COMPLETED`

## 10. 为什么第一版先做单体，不直接做 Spring Cloud

因为当前阶段直接上微服务，会同时引入很多新问题：

- 服务拆分是否合理
- 接口调用方式
- 配置中心
- 服务注册发现
- 网关
- 链路调试
- 分布式事务
- 本地多服务调试成本

这些问题本身就足以把学习重心从“业务实现”拉走。

所以更合理的路线是：

### 第一步

先做一个边界清楚的模块化单体电商系统。

### 第二步

在单体里提前保持上下文边界：

- 认证用户
- 商品目录
- 购物车
- 订单

### 第三步

等业务闭环稳定后，再把它逐步拆成 Spring Cloud 微服务。

## 11. 未来 Spring Cloud 演进方向

后续可以按业务上下文拆成：

### 1. `gateway-service`

统一入口，处理路由和鉴权前置逻辑。

### 2. `user-service`

负责：

- 注册
- 登录
- 用户资料

### 3. `product-service`

负责：

- 分类
- 商品
- 库存

### 4. `cart-service`

负责：

- 购物车项管理

### 5. `order-service`

负责：

- 创建订单
- 查询订单
- 订单状态流转

### 6. 基础设施组件

后续再按需要引入：

- Nacos
- OpenFeign
- Gateway
- Sentinel
- Seata

但这些不进入第一版。

## 12. 当前阶段最值得学习的点

这次电商改造里，你最该重点练的不是“多写几个接口”，而是：

### 1. 业务建模

能不能把商品、购物车、订单之间的关系设计清楚。

### 2. 事务设计

能不能把“扣库存 + 建订单 + 写订单项”放进正确事务边界。

### 3. 状态思维

能不能给订单设计出合理状态，而不是只会做静态 CRUD。

### 4. 资源边界

能不能区分：

- 哪些资源是主资源
- 哪些资源是从属资源
- 哪些接口是用户行为接口

### 5. 微服务前置边界意识

虽然第一版不拆服务，但代码结构和职责划分要尽量避免以后拆分时大面积重写。

## 13. 第一版成功标准

如果第一版完成后，你能做到下面这些，就说明路线是对的：

1. 能完整讲清业务链路
2. 能演示从登录到下单的闭环
3. 能解释为什么订单项要做快照
4. 能解释为什么下单要用事务
5. 能解释后面为什么适合拆成微服务

这时这个项目就已经不再只是 Spring Boot 学习样例，而是一个有明确业务主线和扩展方向的项目。
