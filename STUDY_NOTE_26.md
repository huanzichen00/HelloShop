# Study Note 26

## 主题

业务设计训练：为 `music-review-site` 重写版做第一版范围拆解

## 1. 这节课解决什么问题

这节课不是写代码，而是练一件更重要的事：

**在重写一个真实项目之前，先把业务和范围拆清楚。**

目标项目是：

`music-review-site`

但当前阶段不适合直接全量照着重写，因为原项目里已经包含：

- Spring Security
- JPA
- 文件上传
- 通知
- 博客
- 游戏玩法
- 前端 React

这些东西一起上，复杂度太高。

所以这节课的重点是：

- 先识别核心业务
- 先判断第一版必须做什么
- 先决定哪些功能明确不做

## 2. 第一版最核心要解决什么问题

重写版第一版的核心，不是“做一个音乐百科站”，而是：

**让用户能围绕专辑浏览内容并发表评论。**

也就是说，它最小必须解决：

- 用户注册登录
- 浏览艺术家和专辑
- 查看专辑详情
- 围绕专辑发表评论
- 查看评论
- 删除自己的评论

这就是第一版的业务主线。

## 3. 第一版最小模块

经过这轮训练，第一版最小模块收成了四个：

### 1. 用户模块

解决谁在注册、登录、发表评论，以及评论归属给谁的问题。

### 2. 艺术家模块

解决专辑归属于哪个艺术家、内容如何组织的问题。

### 3. 专辑模块

解决网站主要展示什么内容，以及评论依附在哪个对象上的问题。

### 4. 乐评模块

解决用户如何围绕专辑发表评论、查看评论和管理自己评论的问题。

## 4. 为什么第一版不是先做“曲目、风格、点赞、回复”

因为第一版最重要的是：

**先把最小闭环做出来。**

像：

- 曲目
- 风格
- 点赞
- 回复
- 通知

这些都不是评论站第一版成立的必要条件。

如果一开始就把这些一起做，项目范围会很容易失控。

## 5. 首页主资源判断

一开始很容易把“评论站”理解成首页主体应该是评论。

但这轮训练后更清楚的结论是：

- 首页主展示对象：专辑
- 网站核心互动行为：乐评

也就是说：

### 主资源

- 专辑

### 核心行为

- 评论专辑

这两者不是一回事。

## 6. 评论模块第一版最小闭环

一开始很容易把：

- 点赞评论
- 回复评论

也当成第一版必须功能。

但这轮训练后，`review` 模块更合理的第一版最小闭环是：

### 1. 发评论

用户能产生核心内容。

### 2. 查看专辑下的评论列表

评论能被消费，而不是发完就结束。

### 3. 删除自己的评论

用户能管理自己产生的内容。

这说明：

**第一版先做“写 + 看 + 管”闭环，不先做增强互动。**

## 7. 第一版最先落地的 5 个接口

这轮训练后，第一批最合理的接口顺序收成了：

### 1. 注册

建立最基础的用户体系。

### 2. 登录

建立用户身份，为后续行为提供前提。

### 3. 专辑列表

先把主资源展示出来。

### 4. 艺术家列表

补齐内容组织骨架。

### 5. 专辑详情

为后续评论模块提供承载对象。

这个顺序背后的原则不是“想到哪个做哪个”，而是：

```text
先身份
再内容
再互动
```

## 8. 表关系意识的关键提升

这轮训练里一个很重要的收获是：

### 一对多关系里，外键放在“多”的那一侧

例如：

#### `artist -> album`

- 一个艺术家有多张专辑
- 外键放在 `album.artist_id`

#### `album -> review`

- 一张专辑有多条评论
- 外键放在 `review.album_id`

#### `user -> review`

- 一个用户有多条评论
- 外键放在 `review.user_id`

这比只记“依赖前者”更具体，因为它已经落到了表设计层。

## 9. 字段设计意识的提升

这轮训练还专门练了一个点：

### `review` 表不冗余存展示名称字段

也就是说，评论表里存：

- `user_id`
- `album_id`

但不直接存：

- `username`
- `album_title`
- `artist_name`

原因是这些字段属于别的表自己的信息，应该通过查询组装，而不是直接冗余进评论表。

这一步很重要，因为它代表你开始从“想到什么字段就塞什么”转向：

**区分自身字段、外键字段和派生展示字段。**

## 11. 当前结论

这节课最重要的不是记住几个接口名字，而是建立了这套思维：

### 1. 先识别核心业务

音乐评论站不是音乐百科站，它的核心是“围绕专辑的评论行为”。

### 2. 第一版先做闭环

先做：

- 浏览内容
- 注册登录
- 发评论
- 看评论
- 删自己的评论

### 3. 先做最小版本，再扩展玩法

点赞、回复、通知、收藏、博客、玩法都可以以后再加。

### 4. 先把业务设计说清，再开始建表和写代码

这是重写真实项目时非常重要的一步。
