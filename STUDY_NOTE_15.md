# Study Note 15

## 主题

阶段性代码整理：分离 Demo 接口与收口 `UserService`

## 1. 为什么这一步有必要

当前项目已经不再是最开始那个“只有几个接口练语法”的小样例了。

现在已经有：

- 用户 CRUD
- 参数校验
- 统一返回
- 全局异常处理
- MyBatis 注解版和 XML 版
- 条件查询、分页、排序
- 事务实验

如果继续一边加功能、一边不整理结构，代码会越来越乱。

所以这一步的目标不是新增功能，而是：

**把学习代码和业务代码分开，把重复逻辑收口。**

## 2. Controller 为什么要分开

当前项目现在已经拆成了 3 个 controller：

### `UserController`

负责正式用户业务接口，例如：

- 创建用户
- 查询列表
- 按 id 查询
- 更新
- 删除
- 条件查询
- 分页
- 条件分页排序

### `DemoController`

负责事务实验接口，例如：

- `/demo/transaction-test`
- `/demo/transaction-self-call-test`
- `/demo/transaction-checked-exception-test`

### `HelloController`

只保留配置相关演示接口：

- `/project-name`
- `/project-info`

这样的好处是：

1. 用户业务接口路径更干净
2. 学习实验接口不会污染正式模块
3. 后面做正式项目时，哪些代码该保留、哪些该删，一眼就能看出来

## 3. 为什么事务实验接口不应该留在 `UserController`

如果把这些接口放在 `/users` 下面：

- `/users/transaction-test`
- `/users/transaction-self-call-test`

语义上会很奇怪。

因为它们不是“用户业务”，而是“事务原理实验”。

所以把它们统一移到：

```text
/demo/*
```

是一个正确的阶段性收口。

## 4. `HelloController` 为什么要删掉早期练习接口

最开始为了学习：

- 返回字符串
- 接收参数
- 返回对象 JSON

写过一些教学接口，例如：

- `/hello`
- `/greet`
- `/user-info`

到当前阶段，这些接口已经没有继续保留的必要。

因为主线已经进入：

- 用户业务接口
- 数据库
- 事务
- 分页排序

所以把 `HelloController` 收到只保留配置相关接口，是合理的。

## 5. `UserService` 为什么要做小重构

随着功能变多，`UserService` 里开始出现重复逻辑。

比较明显的有两类：

### 分页参数校验重复

例如：

- `page` 不能为空且要大于 0
- `size` 不能为空且要大于 0

这类逻辑会在多个分页方法里重复。

### 排序参数默认值重复

例如：

- `sort` 为空时默认 `id`
- `order` 为空时默认 `asc`

如果每个方法都各写一遍，后面会越来越乱。

## 6. 当前抽出来的私有方法

为了让 service 主流程更短，当前项目把重复逻辑收成了私有方法，例如：

- `validatePageParams(...)`
- `normalizeSort(...)`
- `normalizeOrder(...)`
- `buildPageResult(...)`

这类方法的作用不是“炫技抽象”，而是：

1. 降低重复
2. 让业务方法更像业务方法
3. 修改规则时只改一个地方

## 7. 一个典型的重构后效果

重构前，分页方法可能同时混着做：

- 参数校验
- 默认值处理
- offset 计算
- mapper 调用
- totalPages 计算
- 返回对象组装

重构后，方法主流程会更像这样：

1. 校验参数
2. 规范化排序参数
3. 计算 offset
4. 查 list
5. 查 total
6. 调 `buildPageResult(...)`

这样的代码更容易读，也更容易维护。

## 8. 这节课真正要记住什么

### 结论 1

学习接口和正式业务接口不要长期混在同一个 controller 里。

### 结论 2

当一个模块开始变大时，应该做阶段性整理，而不是只继续堆功能。

### 结论 3

发现重复逻辑时，要做小重构，但不要一下子引入过度复杂的抽象。

### 结论 4

好的结构不是一开始就完美设计出来的，而是在功能逐步变多后持续收口出来的。

## 9. 当前进度定位

到这一步，你已经开始具备一个很重要的工程意识：

**不是只会“把功能写出来”，而是开始知道什么时候该收结构、怎么收结构。**

这对后面做真正项目很重要。
