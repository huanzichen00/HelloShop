# Music Review Site 重写版第一版设计说明

## 1. 项目定位

这个重写版项目的目标，不是机械复刻现有 `music-review-site` 的全部功能，而是先做一个**可控、可讲、可继续扩展**的第一版后端。

第一版最核心要解决的问题是：

- 用户能够注册和登录
- 用户能够浏览艺术家和专辑内容
- 用户能够围绕专辑发表乐评
- 用户能够查看专辑下的评论，并管理自己的评论

也就是说，第一版要先把：

```text
内容浏览 + 用户认证 + 评论互动
```

这条最小业务闭环做稳。

这个版本先不追求玩法复杂、模块很多，而是优先保证：

- 业务主线清楚
- 技术栈可控
- 后续容易扩展

## 2. 技术路线

### 后端

- Spring Boot
- MyBatis
- MySQL
- JWT
- Redis
- Swagger
- `@Valid` 参数校验
- 统一返回结构
- 全局异常处理

### 当前明确不采用

- Spring Data JPA
- Spring Security

原因是当前阶段更重要的是：

**先把业务重写成功，而不是同时切两条陌生技术线。**

### 前端

前端后续计划使用 Vue，但第一版先不进入前端实现，先把后端业务主线跑通。

## 3. 第一版最小模块

### 1. 用户模块

解决用户注册、登录、当前用户信息以及评论归属问题。

### 2. 艺术家模块

作为内容骨架，解决专辑归属于哪个艺术家、内容如何组织的问题。

### 3. 专辑模块

作为主资源，解决网站主要展示什么内容，以及评论依附在哪个对象上的问题。

### 4. 乐评模块

作为核心互动模块，解决用户如何围绕专辑发表评论、查看评论和管理自己评论的问题。

## 4. 第一版接口类别

第一版接口分成三类：

### 1. 内容浏览类

用于让用户浏览：

- 艺术家
- 专辑
- 专辑详情

### 2. 用户认证类

用于完成：

- 注册
- 登录
- 当前用户识别

### 3. 评论互动类

用于完成：

- 发评论
- 看评论
- 删自己的评论

## 5. 第一批最先落地的 5 个接口

### 1. 注册

用于创建用户数据，建立网站最基础的用户体系。

### 2. 登录

用于完成身份认证，为后续评论等用户行为提供前提。

### 3. 专辑列表

用于展示网站的主体内容，让用户先有可浏览的核心资源。

### 4. 艺术家列表

用于补齐内容组织骨架，让专辑与艺术家之间的关系清晰可见。

### 5. 专辑详情

用于承载专辑的完整展示，并为后续乐评模块提供自然的依附对象。

## 6. 第二批接口建议

在第一批稳定之后，再继续做：

### 1. 查看某张专辑下的评论列表

### 2. 创建评论

### 3. 删除自己的评论

### 4. 当前用户信息

### 5. 修改当前用户资料

这时评论模块才真正形成闭环：

- 写出来
- 看得到
- 自己能管理

## 7. 第一版明确暂不实现

第一版暂不做：

- 点赞评论
- 回复评论
- 收藏
- 通知
- 博客
- 游戏化玩法
- Question Bank
- 在线房间
- 文件上传

原因是这些功能都依赖基础主线已经稳定成立。

如果在第一版就同时加入这些增强互动和扩展玩法，很容易让项目范围失控，影响主线落地。

所以第一版先只做：

```text
浏览内容 + 注册登录 + 发评论 + 看评论 + 删自己的评论
```

后面再逐步扩展。

## 8. 核心资源判断

### 内容主资源

- `artist`
- `album`

### 行为主体

- `user`

### 从属资源

- `review`

这个划分决定了后续设计顺序：

1. 先立住身份和内容骨架
2. 再补围绕内容的用户互动

## 9. 核心表关系

### 1. `artist` 和 `album`

- 一个艺术家对应多张专辑
- 外键放在 `album`
- 即：
  ```text
  album.artist_id
  ```

### 2. `album` 和 `review`

- 一张专辑对应多条评论
- 外键放在 `review`
- 即：
  ```text
  review.album_id
  ```

### 3. `user` 和 `review`

- 一个用户可以写多条评论
- 外键放在 `review`
- 即：
  ```text
  review.user_id
  ```

一对多关系里，外键放在“多”的那一边，因为“多”的那一侧依赖“一”的那一侧存在。

## 10. 第一版核心表最小字段思路

### `user`

建议最小字段：

- `id`
- `username`
- `password`
- `nickname`
- `created_at`

### `artist`

建议最小字段：

- `id`
- `name`
- `country`
- `formed_year`
- `image_url`

### `album`

建议最小字段：

- `id`
- `title`
- `artist_id`
- `release_year`
- `cover_url`

### `review`

建议最小字段：

- `id`
- `user_id`
- `album_id`
- `content`
- `created_at`

第一版 `review` 表里不冗余保存：

- `username`
- `album_title`
- `artist_name`

这些都应该通过关联关系和查询组装得到，而不是直接塞进评论表。

## 11. 第一版实现顺序

如果从零开始重写，推荐顺序是：

### 1. 用户模块

- 注册
- 登录
- JWT
- 当前用户

### 2. 艺术家模块

- 列表
- 详情

### 3. 专辑模块

- 列表
- 详情
- 和艺术家的关系

### 4. 乐评模块

- 专辑下评论列表
- 发评论
- 删除自己的评论

这个顺序的本质是：

```text
先身份
再内容
再互动
```

## 12. 为什么这样重写最合理

因为当前目标不是“最像原项目”，而是：

- 让你真的能独立重写成功
- 保留音乐评论站最核心的业务特征
- 控制第一版复杂度
- 为后面继续扩展打基础

所以这个第一版设计的核心原则就是：

**先做最小闭环，不先做花样。**
