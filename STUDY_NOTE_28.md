# Study Note 28

## 主题

电商系统第一版数据库设计：为什么要这样拆表，为什么索引要这样建

## 1. 这节课解决什么问题

前面已经完成了：

- 电商第一版范围拆解
- `category / product / cart` 主线设计
- Flyway 数据库版本管理

这节课继续往前走，重点解决：

**电商系统第一版的数据库到底怎么设计，为什么这些索引要这样写。**

这一步很重要，因为数据库设计不是“把字段拼出来”就结束了，还要能解释：

- 为什么要拆成这些表
- 为什么有些字段必须保留
- 为什么有些地方加唯一约束
- 为什么索引不是越多越好

## 2. 第一版最核心的 6 张表

这一版数据库最终收成了 6 张核心表：

- `users`
- `categories`
- `products`
- `cart_items`
- `orders`
- `order_items`

这 6 张表刚好对应第一版的业务主线：

```text
注册登录
-> 商品浏览
-> 购物车
-> 下单
-> 我的订单
```

## 3. 为什么要区分内容、过程、结果三类资源

这一轮设计里，一个非常重要的意识提升是：

**不是所有表都属于同一种资源。**

### 内容资源

- `categories`
- `products`

它们解决的是：

```text
卖什么
```

### 过程资源

- `cart_items`

它解决的是：

```text
准备买什么
```

### 结果资源

- `orders`
- `order_items`

它们解决的是：

```text
已经买了什么
```

这个划分很重要，因为它会直接影响：

- 表怎么拆
- 接口怎么排优先级
- 哪些数据应该长期保留
- 哪些数据只是临时过程数据

## 4. 为什么购物车和订单不能混在一张表里

这是第一版建模里很容易混掉的点。

### `cart_items`

是临时过程数据。

特点是：

- 会频繁改数量
- 会切换选中状态
- 可能被删除
- 下单后本身就不再是最终结果

### `orders` / `order_items`

是交易结果数据。

特点是：

- 一旦创建，就变成历史记录
- 后面更多是状态流转，而不是随便改内容
- 需要长期保留

所以这两个概念不能混。

如果把购物车直接当订单用，后面会很难解释：

- 哪些是还没买的
- 哪些是已经下单的
- 哪些字段是临时状态
- 哪些字段是交易结果

## 5. 为什么订单一定要拆成 `orders` 和 `order_items`

这是数据库设计里最典型的一对多场景。

### `orders`

保存一笔订单的主信息：

- 谁下的单
- 订单号是什么
- 总金额多少
- 总件数多少
- 当前状态是什么

### `order_items`

保存这笔订单里的每个商品明细：

- 买了哪件商品
- 下单时商品名是什么
- 下单时单价是多少
- 买了几件
- 这一项小计是多少

如果不拆，直接把多件商品都塞进 `orders`，就会很快失控。

所以这一轮我把它明确成：

```text
orders = 订单主表
order_items = 订单明细表
```

## 6. 为什么 `order_items` 必须做商品快照

这是这轮数据库设计里最值得讲的点之一。

`order_items` 里不只存 `product_id`，还要存：

- `product_name`
- `product_price`
- `product_cover_url`

原因是：

**订单是交易结果，不能依赖当前商品表实时回填。**

例如：

### 商品改名

用户买的时候商品名叫：

```text
iPhone 15 128G
```

后来商品被改成：

```text
iPhone 15 新版
```

如果订单详情总是查 `products.name`，那老订单看到的就不是当时买的那个名字了。

### 商品改价

用户下单时价格是：

```text
4999.00
```

后来涨价到：

```text
5499.00
```

如果订单项不保存 `product_price`，历史订单金额就解释不清。

所以订单快照本质上是在保护：

- 历史一致性
- 交易可追溯性

## 7. 为什么 `cart_items` 要加 `(user_id, product_id)` 唯一约束

这个约束有两个作用。

### 1. 保证数据正确

同一个用户购物车里，同一个商品只能出现一行。

### 2. 支撑“重复添加时累加数量”的业务逻辑

加购物车时，后端可以先按：

```text
(user_id, product_id)
```

查是否已有记录：

- 有，就累加数量
- 没有，再插入新行

如果没有这个唯一约束，数据库层面就挡不住重复数据。

## 8. 为什么 `orders` 要有 `order_no`

`orders.id` 和 `orders.order_no` 不是重复字段，它们职责不一样。

### `id`

更适合数据库内部关联。

比如：

- `order_items.order_id`
- 主键自增

### `order_no`

更适合业务侧使用。

比如：

- 对外展示订单号
- 后续做支付单号关联
- 按订单号查询

所以这一轮设计里，订单必须把：

```text
数据库主键
```

和

```text
业务订单号
```

区分开。

## 9. 这轮数据库索引是怎么想的

这一轮一个很关键的训练点是：

**索引不是越多越好，而是要围绕真实查询路径来建。**

我的索引思路是：

- 唯一约束优先解决数据正确性
- 普通索引优先服务高频查询
- 复合索引优先贴近“过滤 + 排序”的真实场景
- 不为了“显得专业”给每个字段都建索引

## 10. 为什么 `orders` 要建这两个索引

### `uk_orders_order_no (order_no)`

这个是唯一索引。

作用：

- 保证订单号唯一
- 支持按订单号查询

这个索引的重点不是性能，而是：

**先保证业务标识不能重复。**

### `idx_orders_user_id (user_id)`

这个索引服务于：

```text
查某个用户的订单
```

典型 SQL 是：

```sql
select *
from orders
where user_id = ?
```

因为“我的订单”一定是高频场景，所以 `user_id` 很自然应该成为索引。

### `idx_orders_user_created_at (user_id, created_at)`

这个复合索引服务于：

```text
查某个用户的订单列表，并按创建时间倒序分页
```

典型 SQL 是：

```sql
select *
from orders
where user_id = ?
order by created_at desc
limit ?, ?
```

为什么写成：

```text
(user_id, created_at)
```

而不是：

```text
(created_at, user_id)
```

因为真实查询路径是：

1. 先按 `user_id` 过滤
2. 再在这个用户自己的订单里按时间排序

这符合复合索引的最左匹配原则。

简单理解就是：

**先缩小到“这个用户的订单集合”，再在这个集合里取最近几条。**

## 11. 为什么 `order_items` 要建这两个索引

### `idx_order_items_order_id (order_id)`

这个索引服务于：

```text
按订单查明细
```

订单详情页最核心的一条 SQL 基本就是：

```sql
select *
from order_items
where order_id = ?
```

如果没有这个索引，每次查订单详情都要扫整张 `order_items` 表。

### `idx_order_items_product_id (product_id)`

这个索引不是下单主链路必须的，但它很适合为后续扩展做准备。

它支持：

- 按商品反查订单项
- 看某个商品出现在多少笔订单里
- 做销量统计和问题追溯

所以它不是“现在必须”，但属于：

**现在成本不高，后续价值比较高**

的一类索引。

## 12. 为什么没有给所有字段都建索引

这一轮数据库设计里，还有一个很重要的结论是：

**索引不是越多越好。**

因为：

- 索引会占空间
- 索引会增加插入和更新成本
- 写多读多的表如果乱加索引，反而会拖慢写入

所以第一版刻意没有给这些字段单独建索引：

- `orders.status`
- `cart_items.selected`
- `order_items.product_name`
- `order_items.product_cover_url`

原因不是这些字段没用，而是：

- 它们不是当前高频过滤条件
- 它们更多是展示字段
- 当前阶段还没有必要为它们付出额外写入成本

## 13. 这轮数据库设计最值得记住的 4 个点

### 1. 购物车是过程数据，订单是结果数据

所以要分表，不要混。

### 2. 一笔订单天然是一对多

所以必须拆 `orders` 和 `order_items`。

### 3. 订单项必须做商品快照

这样历史订单才不会被商品后续变更污染。

### 4. 索引要围绕真实查询路径设计

不是每个字段都加索引，而是优先给：

- 唯一性约束
- 高频过滤条件
- 高频排序路径

建索引。
