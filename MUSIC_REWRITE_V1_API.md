# Music Review Site 重写版第一版接口设计说明

## 1. 目标

这份文档用于明确 `music-review-site` 重写版第一版到底先做哪些接口、为什么先做这些接口，以及哪些接口可以放到第二批。

核心原则是：

**第一版先把最小业务闭环做稳，不先追求接口数量。**

## 2. 接口分类

第一版接口分成四类：

### 1. 用户认证类

用于建立用户体系和登录态。

### 2. 当前用户类

用于识别当前登录用户，支撑前端登录后的用户状态管理。

### 3. 内容浏览类

用于让用户浏览艺术家和专辑。

### 4. 评论互动类

用于让用户围绕专辑产生和管理评论内容。

## 3. 第一批必须做的接口

这些接口属于：

**没有它们，第一版业务闭环立不住。**

### 1. `POST /auth/register`

作用：

- 创建用户
- 建立最基础的用户体系

为什么第一批必须做：

因为没有注册，网站就没有真实用户来源。

### 2. `POST /auth/login`

作用：

- 完成登录
- 返回 token

为什么第一批必须做：

因为评论这类用户行为必须建立在身份识别之上。

### 3. `GET /auth/me`

作用：

- 返回当前登录用户信息

为什么第一批必须做：

因为前端登录后通常需要先知道“当前是谁”，这会直接影响页面状态和权限判断。

### 4. `GET /artists`

作用：

- 展示艺术家列表

为什么第一批必须做：

因为艺术家是内容骨架，用来组织专辑归属。

### 5. `GET /albums`

作用：

- 展示专辑列表

为什么第一批必须做：

因为专辑是音乐评论站第一版的主资源，也是网站主体内容入口。

### 6. `GET /albums/{id}`

作用：

- 查看专辑详情

为什么第一批必须做：

因为评论需要依附在专辑详情语义上，没有专辑详情页，评论模块缺少自然落点。

### 7. `GET /albums/{id}/reviews`

作用：

- 查看某张专辑下的评论列表

为什么第一批必须做：

因为评论必须能被消费，而不是只能写不能看。

### 8. `POST /albums/{id}/reviews`

作用：

- 针对某张专辑发表评论

为什么第一批必须做：

因为这是评论站最核心的用户行为。

### 9. `DELETE /reviews/{id}`

作用：

- 删除自己的评论

为什么第一批必须做：

因为第一版要保证用户对自己产生的内容有基本管理能力。

## 4. 第二批再做的接口

这些接口有价值，但不是第一版最小闭环必须项。

### 1. `PUT /users/me`

作用：

- 修改当前用户资料

为什么放第二批：

它很实用，但优先级低于“先建立评论闭环”。

### 2. `GET /artists/{id}`

作用：

- 查看艺术家详情

为什么放第二批：

第一版有艺术家列表和专辑归属已经足够支撑内容骨架，艺术家详情可以后补。

### 3. 收藏相关接口

例如：

- `POST /albums/{id}/favorite`
- `DELETE /albums/{id}/favorite`
- `GET /users/me/favorites`

为什么放第二批：

收藏是增强用户行为，不影响评论站第一版成立。

### 4. 回复评论相关接口

例如：

- 创建回复
- 查看回复列表

为什么放第二批：

回复属于评论之上的二级互动，应该在评论主线稳定之后再补。

### 5. 点赞评论相关接口

为什么放第二批：

它提升互动性，但不属于第一版评论最小闭环。

### 6. 通知相关接口

为什么放第二批：

通知依赖更多互动行为先稳定存在。

## 5. 为什么不是所有模块都做完整 CRUD

因为不同模块的业务角色不一样。

### `review`

更接近用户行为型资源，所以第一版自然需要：

- 创建
- 列表
- 删除自己的内容

### `artist`

更接近内容库型资源，所以第一版通常只需要：

- 列表
- 可能后补详情

而不是让普通用户随便创建和删除艺术家。

### `album`

作为主资源，第一版最小闭环更关注：

- 列表
- 详情

而不是普通用户直接创建专辑。

## 6. 第一版接口顺序背后的原则

接口推进顺序不是随机的，而是遵守：

```text
先身份
再内容
再互动
```

也就是：

### 第一步

先让系统知道：

- 谁在使用
- 谁能登录

### 第二步

再让用户有内容可看：

- 艺术家
- 专辑

### 第三步

再围绕内容开放互动：

- 看评论
- 发评论
- 删自己的评论

## 7. 第一版最小业务闭环

第一版真正要成立的闭环是：

1. 用户注册登录
2. 浏览专辑
3. 进入专辑详情
4. 查看评论列表
5. 发表评论
6. 删除自己的评论

如果这 6 步完整成立，这个网站第一版就已经站住了。

## 8. 当前结论

这份第一版接口设计说明的核心不是“列更多接口”，而是明确：

- 什么是第一批必须做
- 什么可以第二批再做
- 为什么 `review` 比点赞/回复更优先
- 为什么 `album` 是主资源
- 为什么 `artist` 不需要第一版就做完整 CRUD

这份文档后面会直接指导：

- DTO 设计
- Controller 设计
- Service 设计
- 接口开发顺序
