# Music Review Site 重写版第一版开发顺序说明

## 1. 目标

这份文档用于明确 `music-review-site` 重写版第一版真正开始实现时，应该按什么顺序落地，避免一开始同时铺太多线，导致项目失控。

当前核心原则只有一句：

```text
先身份，再内容，再互动
```

也就是：

1. 先把用户认证主线做通
2. 再把艺术家和专辑内容主线做通
3. 最后再补评论互动主线

## 2. 为什么不先把所有表一次建完

因为当前项目的目标不是“最快把所有 SQL 写出来”，而是：

- 一步步建立可运行的系统
- 每完成一段就能调通一段
- 每一段都能单独验证

如果一开始把所有表、所有接口、所有模块一起铺开，会有几个问题：

- 错误定位困难
- 模块依赖混乱
- 很容易越做越散

所以更合理的做法是：

**按业务主线分阶段推进。**

## 3. 总体开发顺序

推荐顺序：

### 第一阶段：项目骨架

- 创建 Spring Boot 项目
- 搭基础目录结构
- 接 MySQL
- 接 MyBatis
- 接统一返回和全局异常
- 接 Swagger

### 第二阶段：认证主线

- `user` 表
- 注册
- 登录
- JWT
- 当前用户

### 第三阶段：内容主线

- `artist` 表
- `album` 表
- 艺术家列表
- 专辑列表
- 专辑详情

### 第四阶段：评论主线

- `review` 表
- 查看专辑评论列表
- 创建评论
- 删除自己的评论

这就是第一版的完整最小闭环。

## 4. 第一阶段：项目骨架

### 先做什么

1. 创建新的 Spring Boot 项目
2. 按 [MUSIC_REWRITE_V1_STRUCTURE.md](/Users/huanzichen/interview/SpringBoot/springboot-hello/MUSIC_REWRITE_V1_STRUCTURE.md) 建好目录
3. 接入：
   - MySQL
   - MyBatis
   - Redis
   - Swagger
4. 准备基础通用类：
   - `Result`
   - `ErrorCode`
   - `BusinessException`
   - `GlobalExceptionHandler`

### 这一阶段的完成标准

- 项目能启动
- 能连数据库
- Swagger 能打开
- 基础异常和统一返回链路已经可用

### 这阶段先不要做什么

- 不要先写业务模块
- 不要先写 JWT
- 不要先写表以外的一堆 SQL 脚本

## 5. 第二阶段：认证主线

### 先建哪张表

先建：

```text
user
```

### 为什么先做用户

因为：

- 注册登录依赖它
- 评论归属依赖它
- 当前用户能力依赖它

### 先做哪些接口

#### 1. 注册

```text
POST /auth/register
```

#### 2. 登录

```text
POST /auth/login
```

#### 3. 当前用户

```text
GET /auth/me
```

### 这一阶段要补哪些基础能力

- BCrypt
- JWT
- `UserContext`
- 登录拦截器

### 这阶段测试重点

- 注册成功
- 登录成功
- 未登录访问受保护接口返回未登录
- 当前用户接口能拿到登录用户

### 这阶段先不要做什么

- 先不要做 `PUT /users/me`
- 先不要做用户头像
- 先不要做复杂权限

## 6. 第三阶段：内容主线

### 先建哪张表

先建：

```text
artist
```

再建：

```text
album
```

### 为什么这个顺序合理

因为：

- `artist` 是内容骨架
- `album` 要依赖 `artist_id`

### 先做哪些接口

#### 1. 艺术家列表

```text
GET /artists
```

#### 2. 专辑列表

```text
GET /albums
```

#### 3. 专辑详情

```text
GET /albums/{id}
```

### 这阶段测试重点

- 艺术家列表能返回数据
- 专辑列表能返回数据
- 专辑详情能按 id 正确返回
- 专辑和艺术家的归属关系正确

### 这阶段先不要做什么

- 先不要做艺术家详情
- 先不要做专辑创建/编辑后台
- 先不要做风格、曲目、搜索

## 7. 第四阶段：评论主线

### 先建哪张表

建：

```text
review
```

### 为什么 `review` 最后做

因为它同时依赖：

- `user`
- `album`

如果前两条主线没立住，评论模块就没有落点。

### 先做哪些接口

#### 1. 查看专辑评论列表

```text
GET /albums/{id}/reviews
```

#### 2. 创建评论

```text
POST /albums/{id}/reviews
```

#### 3. 删除自己的评论

```text
DELETE /reviews/{id}
```

### 这阶段测试重点

- 能查到某张专辑下的评论列表
- 登录用户能创建评论
- 未登录用户不能创建评论
- 用户只能删除自己的评论

### 这阶段先不要做什么

- 先不要做编辑评论
- 先不要做点赞
- 先不要做回复
- 先不要做通知

## 8. 第一版完成标准

如果下面这些都成立，就说明重写版第一版已经站住了：

### 用户体系

- 能注册
- 能登录
- 能识别当前用户

### 内容体系

- 能查看艺术家列表
- 能查看专辑列表
- 能查看专辑详情

### 评论体系

- 能查看评论列表
- 能创建评论
- 能删除自己的评论

### 技术基础

- 有统一返回
- 有全局异常处理
- 有 JWT 鉴权
- 有 Swagger

## 9. 第一版明确不做什么

在第一版开发顺序里，明确先不做：

- 收藏
- 回复
- 点赞
- 通知
- 文件上传
- 博客
- 游戏玩法
- Question Bank
- 复杂搜索

原因不是这些没价值，而是：

**它们都应该建立在前面的主线已经稳定成立之后。**

## 10. 当前结论

重写版第一版后端最合理的落地顺序是：

```text
项目骨架
-> 用户认证
-> 艺术家与专辑
-> 评论模块
```

这个顺序的本质是：

- 先让系统能识别用户
- 再让用户有内容可看
- 最后再让用户对内容产生互动

这会比“一开始什么都一起做”稳定得多，也更符合你当前阶段的能力结构。
