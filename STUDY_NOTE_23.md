# Study Note 23

## 主题

任务状态变更记录 `task_status_logs`

## 1. 这节课解决什么问题

前面的 `tasks` 模块已经有了：

- 状态字段 `status`
- 完成时间 `completedAt`
- 状态切换时自动维护完成时间

但如果系统里只是“当前状态是什么”，还不够。

真实项目里经常还需要知道：

- 任务原来是什么状态
- 后来改成了什么状态
- 是谁改的
- 什么时候改的

所以这节课新增的是：

**任务状态变更记录**

也就是：

```text
task_status_logs
```

## 2. 为什么状态变更记录比普通日志更有价值

因为它不是泛泛的“打印一条日志”，而是直接围绕业务对象建了一张表。

这张表记录的是：

- 某个任务
- 从什么状态
- 变成什么状态
- 操作人是谁
- 发生时间是什么时候

这会让项目从“有状态字段”继续升级成“有状态历史”。

也就是说：

**项目开始具备可追踪的业务变化记录。**

## 3. 表结构设计

当前最小版本的表结构是：

```sql
task_status_logs
```

字段包括：

- `id`
- `task_id`
- `old_status`
- `new_status`
- `operator_user_id`
- `created_at`

这是一个很收敛的设计，因为当前只记录：

**任务状态变化**

而不去做过度复杂的“全字段审计日志”。

## 4. 为什么先不做通用操作日志

因为通用日志虽然听起来更高级，但当前阶段更容易失控。

比如一旦做成“通用操作日志”，你就要处理：

- 不同模块
- 不同操作类型
- 不同字段变化
- 序列化前后值

这会明显增加复杂度。

而状态变更记录的好处是：

- 业务语义强
- 实现简单
- 项目价值高
- 很适合实习项目讲解

所以当前阶段先做“任务状态变更记录”是更合理的。

### 2. 新建 mapper

当前阶段只做了最小插入能力：

```java
insert(TaskStatusLog taskStatusLog)
```

也就是说，这节课先打通：

**写日志**

还没有继续做“查日志列表”。

### 3. 在 `TaskService#updateTask` 里挂日志

核心逻辑是：

1. 更新前先拿到旧状态
2. 更新任务
3. 更新后拿到新状态
4. 只有状态真的发生变化，才写一条日志

这说明：

**不是每次更新任务都写日志，而是只有状态变化才写。**

## 6. 这节课踩到的关键 bug

一开始在 `updateTask(...)` 里写成了：

```java
String oldStatus = taskUpdateDTO.getStatus();
```

这其实拿到的是：

**前端传来的新状态**

不是数据库里的旧状态。

结果后面比较时变成了：

```java
新状态 vs 新状态
```

当然永远相等，所以永远不会写日志。

正确写法应该是：

```java
String oldStatus = taskInfo.getStatus();
```

也就是：

**旧状态必须从数据库查出来的原任务对象里拿。**

这个 bug 很有代表性，因为它体现了一个重要原则：

在做“变更记录”时，一定要分清：

- 旧值来自哪里
- 新值来自哪里

## 7. 当前记录规则

现在任务状态变更记录的规则是：

### 会记日志

- `TODO -> DONE`
- `DONE -> TODO`

### 不会记日志

- `TODO -> TODO`
- `DONE -> DONE`

也就是：

**只有状态真的变化了，才写日志。**

这比每次更新都写一条更合理，因为它能避免无意义日志堆积。

## 8. 日志里记录了什么

当前一条状态变更日志至少能表达：

- 这是哪条任务
- 原来是什么状态
- 现在是什么状态
- 是哪个用户操作的
- 操作发生在什么时候

这意味着后面如果你继续扩展，很自然就能做：

- 查看某个任务的状态变更历史
- 查看某个用户最近做过哪些状态操作

## 9. 这一节的价值

这节课完成后，任务模块又提升了一层：

- 不只有当前状态
- 还有状态变化历史
- 不只有最终结果
- 还有变化过程记录

这类设计比继续堆平面接口更像真实业务系统。

在面试里你现在已经可以讲：

- 我不只做了任务状态字段
- 还记录了状态变更日志
- 可以追踪是谁在什么时候把任务从什么状态改成了什么状态

这比“我会写 CRUD”明显更有说服力。
