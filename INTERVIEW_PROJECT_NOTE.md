# Interview Project Note

## 1 分钟项目介绍版

我现在在做一个基于 Spring Boot、MyBatis、MySQL 和 Redis 的后端练习项目，项目里目前有 `users` 和 `tasks` 两个业务模块。用户模块已经实现了完整的 CRUD、参数校验、统一返回、全局异常处理、分页、排序和多条件动态查询；任务模块也完成了基础 CRUD 和条件分页查询。除此之外，项目现在还补了注册、用户名密码登录、JWT 鉴权、Redis 黑名单 logout、当前用户资料维护以及任务的数据权限控制。

这个项目里我重点练了几个后端核心能力。第一是基于 DTO、Service、Mapper 的分层开发；第二是用 MyBatis 注解版和 XML 动态 SQL 分别处理固定 SQL 和复杂查询；第三是统一返回结构、业务异常和参数校验这套接口规范；第四是事务相关实验，包括事务回滚、同类调用失效以及 `rollbackFor` 的影响。另外，现在 `users` 和 `tasks` 两个模块之间也已经做了基础关联查询。现在我在继续把这个项目往更完整的小型后端项目方向推进。

## 3 分钟项目讲解版

### 1. 项目是做什么的

这是一个我用来系统练习 Spring Boot 后端开发的项目。项目目前主要围绕两个模块展开：

- `users`：用户管理模块
- `tasks`：任务管理模块

我做这个项目的目标不是只把接口跑通，而是把后端开发里比较常见的一整套能力串起来，比如分层、数据库访问、参数校验、统一返回、异常处理、分页查询和事务。

### 2. 我做了哪些功能

在 `users` 模块里，我已经实现了：

- 用户的增删改查
- 按 id 查询单个用户
- 按姓名模糊查询
- 分页查询
- 条件分页查询
- 排序查询
- 多条件动态查询
- 查询当前登录用户信息
- 修改当前登录用户资料
- 查询当前登录用户自己的任务列表

在 `tasks` 模块里，我实现了：

- 任务的增删改查
- 按 id 查询单个任务
- 按标题、状态、用户 id 条件分页查询
- 当前登录用户任务状态统计
- 任务状态变更记录
- 查询任务状态变更记录列表
- 登录后只允许查看自己的任务
- 登录后只允许更新和删除自己的任务
- 创建任务时自动绑定当前登录用户
- 任务状态变成 `DONE` 时自动记录完成时间
- 任务恢复成 `TODO` 时自动清空完成时间

另外，这两个模块之间已经有了基础关联：

- 可以根据用户查询任务列表
- 也可以根据任务反查所属用户

在接口层面，我把成功和失败返回统一成了 `Result<T>`，分页接口统一成了 `PageResult<T>`，这样接口结构更规范，也更接近真实项目。

### 3. 我重点解决了哪些技术问题

这个项目里我重点练了 5 类技术点。

第一，分层开发。  
我把代码拆成了 DTO、Controller、Service、Mapper、Model 几层，Controller 负责接收请求，Service 负责业务逻辑，Mapper 负责和数据库交互。

第二，MyBatis 的两种用法。  
简单固定 SQL，比如基础 CRUD，我用注解版 MyBatis；复杂查询，比如分页、排序、多条件动态查询，我切到了 XML 版，用 `<where>`、`<if>`、`<choose>` 来做动态 SQL。

第三，统一接口规范。  
我做了统一返回结构 `Result<T>` 和 `PageResult<T>`，同时实现了 `BusinessException`、`ErrorCode` 和全局异常处理器，这样业务异常和参数异常都能统一返回给前端。

第四，参数校验。  
我在 DTO 层用了 `@Valid`、`@NotBlank`、`@NotNull`、`@Min` 这些校验注解，把很多非法请求拦在接口入口，同时通过全局异常处理把默认错误结果收口成自己的 JSON 格式。

第五，事务。  
我专门做了几个事务实验，验证了：

- `RuntimeException` 默认会回滚
- checked exception 默认不会回滚
- `rollbackFor = Exception.class` 会改变回滚规则
- 同一个类内部直接调用事务方法时，事务可能失效

这些实验让我不是只会用 `@Transactional`，而是开始理解它的生效条件和常见坑。

第六，最小鉴权和权限控制。  
我先用 `userId` 模拟登录，登录成功后生成 token，并通过拦截器保护 `/tasks/**`。在任务模块里，我又进一步限制了数据权限，让登录用户只能查看和操作自己的任务。这一块让我开始接触“登录”和“数据权限”两层不同的后端能力。

第七，JWT 和 Redis 黑名单。  
后面我把最初的 UUID + 内存 token 表升级成了 JWT，这样 token 本身就能携带用户身份。然后我又接入了 Redis，用黑名单机制解决了“JWT logout 后仍然有效”的问题。这一块让我开始理解 JWT 的无状态特点，以及为什么实际项目里 logout 往往需要 Redis 这样的额外状态配合。

第八，任务业务规则。  
后面我没有让 `tasks` 模块停在基础 CRUD，而是继续加了状态流转规则。比如任务是 `TODO` 时不应该有完成时间，变成 `DONE` 时后端会自动写入 `completedAt`，如果再改回 `TODO`，又会自动清空这个时间。同时，任务归属也不再允许通过更新接口随意修改。这一块让我开始意识到，项目里有些字段应该由后端根据业务规则维护，而不是完全交给前端。

第九，当前用户主线。  
我后面又补了 `/auth/me`、`PUT /users/me` 和 `GET /users/me/tasks`。这样项目里不只是“能登录”和“能查任务”，而是开始具备当前登录用户查询自己资料、修改自己资料、查看自己任务的完整主线。这一块让我开始从“认证接口”进一步走向“用户中心型接口设计”。

第十，任务概览接口。  
后面我又补了 `GET /tasks/status-summary`，专门返回当前登录用户任务的状态统计，比如 `TODO` 和 `DONE` 各有多少个。这个接口不是传统 CRUD，而是更偏首页概览和业务视图，这让我开始接触“统计型接口”和“仪表盘接口”的设计思路。

第十一，任务状态变更记录。  
后面我又给任务模块加了状态变更日志表，在任务状态真的发生变化时，会记录任务 id、旧状态、新状态、操作人和操作时间。这样项目里不只知道“任务现在是什么状态”，还知道“任务状态是怎么一步步变化过来的”。这让我开始接触业务变更记录和更贴近真实系统的审计思路。

第十二，状态日志查询。  
在写入状态变更日志之后，我又补了按任务查询状态日志的接口，而且它也复用了任务归属权限。也就是说，只有任务拥有者自己才能查看这条任务的状态变化历史。这让我开始意识到，权限控制不只是保护主资源，也应该保护围绕主资源派生出来的历史记录和统计数据。

## 最值得讲的 5 个技术点

### 1. 统一返回结构

我自己设计了 `Result<T>` 和 `PageResult<T>`，把普通接口和分页接口的返回风格统一了。这样前端调用时更稳定，后面做异常处理和接口说明也更清晰。

### 2. 全局异常处理

我通过 `BusinessException`、`ErrorCode` 和 `GlobalExceptionHandler`，把业务异常和参数校验异常统一转换成了规范 JSON，而不是直接暴露 Spring 默认错误页面。

### 3. MyBatis 注解版与 XML 版的分工

我没有一开始就全用 XML，而是先用注解版做简单 CRUD；当遇到排序、分页、多条件查询这些动态 SQL 场景时，再切换到 XML。这样我对两种写法的适用场景有更清楚的理解。

### 4. 分页和多条件动态查询

我实现了分页接口，并进一步扩展到条件分页和排序分页。后面又用 XML 的 `<where>` 和 `<if>` 实现了多条件动态查询，比如同时按姓名和年龄范围筛选用户。

### 5. 事务实验

我不只是用了 `@Transactional`，还通过实验验证了回滚规则、同类内部调用失效、`rollbackFor` 等事务机制。这一块很适合在面试里讲，因为它能体现你不只是会背注解，而是做过验证。

### 6. 最小登录鉴权与数据权限

我在项目里先做了一个最小登录系统，登录接口通过 `userId` 模拟登录，成功后签发 token。然后我用 Spring MVC 拦截器保护了 `/tasks/**`，未登录不能访问任务接口。再往前走一步，我没有停在“能登录”这一层，而是在任务模块里加了基础数据权限，让登录用户只能查看和操作自己的任务，访问别人的任务会返回 `403 no permission`。

### 7. JWT 与 Redis 黑名单 logout

后面我把最初的随机 token + 内存 `Map` 升级成了 JWT。登录成功后，服务端签发 JWT，拦截器直接解析 JWT 获取当前用户 id，而不是再依赖内存映射。升级成 JWT 后，我又遇到了一个典型问题，就是 logout 后 token 仍然还能继续用。为了解决这个问题，我接入了 Redis，把 logout 的 token 放进黑名单，在拦截器里先查 Redis，再决定是否放行。这样 JWT 的 logout 才真正生效。

## 电商改造里可直接讲的数据库设计点

如果后面面试里要讲这个项目往电商系统升级的部分，我会重点讲 4 个数据库设计点。

### 1. 为什么要拆 `cart_items` 和 `orders`

因为购物车是过程数据，订单是交易结果数据，两者生命周期和业务含义完全不同。购物车会频繁修改和删除，而订单一旦创建就应该作为历史记录长期保留，所以不能混成一张表。

### 2. 为什么订单要拆 `orders` 和 `order_items`

一笔订单天然是一对多关系。`orders` 负责承载订单主信息，比如用户、订单号、总金额、状态；`order_items` 负责承载订单里的商品明细。这样结构更符合真实交易模型，也方便后面扩展订单详情、支付和发货。

### 3. 为什么 `order_items` 要做商品快照

我在订单项里不会只存 `product_id`，还会保存下单时的 `product_name`、`product_price`、`product_cover_url`。原因是订单是交易结果，不应该受后续商品改名、改价影响，这样历史订单才可追溯。

### 4. 为什么索引这样设计

索引不是越多越好，而是围绕真实查询路径设计。比如购物车表用 `(user_id, product_id)` 唯一约束保证同一用户购物车里同一商品只有一行；订单表给 `order_no` 做唯一索引保证业务订单号唯一，再给 `(user_id, created_at)` 做复合索引服务“我的订单列表按时间倒序分页”；订单项表给 `order_id` 建索引服务订单详情查询。这些索引都直接对应真实业务访问路径，而不是为了显得专业随便加。
